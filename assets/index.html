Q1 -- SVN Inquiry

Is there an integration with SVN? We are using SVN externals; if we decided to migrate from SVN to Git, how would we go about doing that?

Answer

Dear Mr. Doe,

Thank you for reaching out about SVN migration. I know this process can be overwhelming at times but it is definitely worth it. Thankfully many developers have gone through the same hardship and have made awesome tools to ease the burden.

The best tool to help you migrate is actually Git itself. Git comes bundled with a command called 'git-svn', which helps you migrate your Subversion repos to Git repos. Since you are using SVN externals you will also need to convert those to Git Submodules.

A high level heuristic for getting this done is as follows:

  • 1) Download Git
  • 2) Retrieve a list of all Subversion authors
  • 3) Clone Subversion repository with 'git svn clone'
  • 4) If external modules, then create Git submodules
  • 5) Push to a Git repository

A great step by step guide on converting SVN to Git was written by Patrick Bougie, a web developer with a ton of contributions to the open source world. He covers migrating from SVN to Git and emulating SVN Externals using Git Submodules. Try following his guide and contact me if you have any questions or run into any issues.

Once you have migrated your SVN code to Git we can set up a call to talk about which GitHub plan will work best for you and your team.

Keep me updated on your migration and let's talk soon!

Q3 -- GitHub.com Server Question

Hi GitHub! My company has an internal Git server and use GitHub.com. For workflow purposes, how do I sync my repository to both GitHub and the internal Git server?

Note: One common option is git remotes.

Answer

Dear Mr. Doe,

Thank you for reaching out about repository syncing! This is a common problem I see businesses running into and thankfully Git makes it easy to solve.

All we have to do is add remote repositories to our local repository, and Git will start pushing to and pulling from each location. Below are the steps used to setup remote repositories from the command line:

$ git remote add repo1 https://192.168.0.255/username/repo1.git
$ git remote set-url --add repo1 https://github.com/username/repo2.git
$ git remote add repo2 https://github.com/username/repo2.git
$ git push -vu repo1 master

Now we can list out all of our remotes:

$ git remote -v
repo1 https://192.168.0.255/username/repo1.git (fetch)
repo1 https://192.168.0.255/username/repo1.git (push)
repo1 https://github.com/username/repo2.git (push)
repo2 https://github.com/username/repo2.git (fetch)
repo2 https://github.com/username/repo2.git (push)

The output shows us that we are now synced to two different remote repositories! Now when we push code the changes will be reflected in both repositories.

Q7 -- Mirroring

We are looking at GitHub Enterprise as a version control and collaboration solution for our company. We have developers scattered across the world and in some of these regions the internet connectivity is terrible, which means git clones or svn checkouts are extremely slow. Does GitHub Enterprise have a way of deploying mirrors in these regions? If so are these read/write mirrors, meaning we also benefit from great improvements on large data pushes?

Answer

Dear Ms. Doe,

Yes I'm glad you asked! GitHub does offer full read/write mirroring capabilities. For your specific situation I would recommend going with our Geo-replication feature.

GitHub Enterprise offers full fledged Geo-replication for all of your company's repositories.

This means configuring a primary repository and replicas in various regions your company does business in. This way when an employee wants to clone a repository, Geo DNS will route the network request to the closet server, so latency is reduced.

With Geo-replication you are harnessing the full power of Git (read and write) and each repository is mirrored across every available replica. Note that data pushes are potentially limited by your least performant replica, but this shouldn't be a problem if each replica has enough resources allocated to it.

I'd love to get some more information from you so we can discuss next steps. Specifically will your company be utilizing the cloud or going with an on-premises installation? And also which areas of the globe is your team located?
